# Contest demo polish: intake shortcut, B2B dashboard, nav lockdown

The change strips the CultureConnect prototype down to a tight demo path for a contest deadline. The landing page intake form now auto-authenticates via the in-memory auth store and navigates directly to the dashboard — no loading animation, no vendor matching, no intermediate screens. The dashboard replaces generic event-count stat cards with hardcoded B2B ROI metrics (87% engagement, +34% connection score, 94% retention, 3.2x ROI) and adds a budget allocation bar chart. Navigation is reduced to brand, Dashboard (when authenticated), and Sign out. Landing copy is repositioned from "book vetted vendors" to "transform corporate event spend into employee retention."

**Watch for:** Dead route files (vendors, analytics, login, signup) remain on disk and are still routable despite being unreachable from the UI (confirmed). The dashboard's unauthenticated guard still links to `/login`, creating an inconsistent escape path if someone hits `/dashboard` directly without auth (confirmed).

## High-level view

The intake form's submit handler collapses from a multi-step vendor-matching simulation into two synchronous calls: `authStore.login()` then `navigate({ to: "/dashboard" })`. Form fields remain for visual credibility during the demo but their values are discarded. HTML `required` attributes still gate submission, so the demo path requires filling fields before proceeding.

The dashboard now tells a static B2B story. The four metrics are string literals pulled from a central `engagementMetrics` object in `data.ts`, and a new budget allocation bar chart draws from `budgetAllocationData`. The "Recommended Vendors" section is removed, decoupling the dashboard from the vendors dataset.

Navigation surgery removes Vendors, Analytics, Sign in, and Get matched links. Nothing renders for unauthenticated users except the brand link. However, the file-based router still registers `/vendors`, `/vendors/:vendorId`, `/dashboard/analytics`, `/login`, and `/signup` as reachable routes that can be accessed by direct URL entry.

## Intake form collapse and auto-auth

The new handler is two lines:

```tsx
authStore.login("demo@acmecorp.com", "demo");
navigate({ to: "/dashboard" });
```

`authStore.login` is synchronous — it mutates module-level state and notifies listeners inline. Navigation fires immediately after. The dashboard picks up auth state via `useSyncExternalStore` on mount, so there's no race between login and render.

The form collects city, team size, date, budget, time slot, and vibes — all `required` via HTML validation — but submits none of it anywhere. The `required` attributes still gate submission, meaning the demo path forces the user to fill fields before proceeding. If the intent is zero-friction demo entry, those attributes would need removal. If the intent is visual realism during a live walkthrough (filling the form is part of the narrative), they're correctly left in place.

## Orphaned routes and nav surface area

TanStack Router's file-based routing auto-registers every file in `src/routes/`. The following routes remain reachable by direct URL:

- `/vendors` and `/vendors/:vendorId` — full vendor listing and detail pages
- `/dashboard/analytics` — deep analytics with pulse surveys, department sentiment, manager scores
- `/login` and `/signup` — auth pages

For a contest demo where judges follow the guided path, this is unlikely to surface. But if anyone explores, they'll find pages that reference links and flows removed from the navbar. The analytics page still has a "Back to Dashboard" link that works, and the vendor pages still link to each other — these form a hidden secondary app that contradicts the streamlined narrative.

The `dashboard.index.tsx` unauthenticated fallback renders `<Link to="/login">Sign in</Link>`. The route exists and works, but it's a path the demo flow never hits because the intake form pre-authenticates before navigating to `/dashboard`.
